# 头脑风暴：将想法转化为设计

> 来源：Superpowers (228k stars) — obra/superpowers/skills/brainstorming
> 原文链接：https://github.com/obra/superpowers/blob/main/skills/brainstorming/SKILL.md

通过自然的协作对话，将想法转化为完整的设计和规格说明。

先了解当前项目上下文，然后一次一个问题地细化想法。一旦理解了要构建什么，呈现设计并获得用户批准。

## 硬门控 (HARD-GATE)

在你呈现设计并获得用户批准之前，**禁止**调用任何实现技能、编写任何代码、搭建任何项目骨架、或采取任何实现行动。这适用于**每个**项目，无论其看起来多么简单。

## 反模式："这太简单了，不需要设计"

每个项目都要经过这个流程。一个待办事项列表、一个单函数工具、一个配置变更——全部都要。"简单"的项目正是那些未经审视的假设导致最多返工的地方。设计可以很短（对于真正简单的项目几句话就行），但你**必须**呈现它并获得批准。

## 检查清单

你**必须**为以下每一项创建任务并按顺序完成：

1. **DoR 检查** — 按以下流程检查需求是否 Ready：
   ```
   a. 读取 features.json 中当前任务的 context_ref
      → 有值 → 直接读取该 PRD
      → 无值 → 搜索 .harness/context/product/ 下匹配的 PRD（按 prd_id 或标题关键词）
      → 找不到 → 提示用户："没有找到对应的 PRD。是否需要：
         (1) 指定已有 PRD 路径
         (2) 创建新 PRD（使用 PRD-TEMPLATE-L 或 PRD-TEMPLATE-M）
         (3) 口头描述需求（S 级，不需要 PRD）"
   
   b. 找到 PRD 后，读取 YAML frontmatter 的 prd_status
      → L 级：检查 product + development + testing 是否全部 done，approved_by 是否有值
      → M 级：检查 product 是否 done，approved_by 是否有值
      → S 级：跳过 DoR 检查
   
   c. DoR 未通过 → 列出缺失项（如："prd_status.development 为 pending，需要 Dev 补充数据模型和 API 契约"）
      → 拒绝进入实现阶段
      → 可以帮助补充技术部分（数据模型、API 契约），但修改后需人类确认
   
   d. DoR 通过 → 继续下一步
   ```
   
   **注意：** PRD 中有 `<!-- agent-load: xxx -->` 标签。Agent 只加载与当前任务相关的部分：
   - `always` → 必须读
   - `backend` → 后端开发时读
   - `frontend` → 前端开发时读
   - `qa` → 测试时读
2. **探索项目上下文** — 检查文件、文档、最近的提交
3. **提出澄清性问题** — 一次一个，理解目的/约束/成功标准
4. **提出 2-3 种方案** — 附带权衡分析和你的推荐
5. **呈现设计** — 按复杂度分段呈现，每段后获得用户批准
6. **写设计文档** — 保存到 `docs/YYYY-MM-DD-<主题>-design.md` 并提交
7. **规格自检** — 快速检查占位符、矛盾、歧义、范围（见下文）
8. **用户审阅规格** — 请用户审阅规格文件后再继续
9. **转入实施** — 调用 plan skill 创建实施计划

## 流程

### 理解想法

- 先查看当前项目状态（文件、文档、最近提交）
- 在提详细问题之前，评估范围：如果需求描述了多个独立子系统，立即指出。不要花时间细化一个需要先分解的项目的细节
- 如果项目太大无法用单个规格覆盖，帮用户分解为子项目：独立的部分有哪些、如何关联、按什么顺序构建？然后对第一个子项目走正常的设计流程
- 对范围合适的项目，一次一个问题地细化想法
- 尽可能用选择题，但开放性问题也可以
- 每条消息只问一个问题——如果一个话题需要更多探索，拆成多个问题
- 聚焦于理解：目的、约束、成功标准

### 探索方案

- 提出 2-3 种不同方案，附带权衡分析
- 以对话方式呈现选项，给出你的推荐和理由
- 以你推荐的选项开头，解释为什么

### 呈现设计

- 一旦你认为理解了要构建什么，呈现设计
- 按复杂度缩放每个部分：简单的几句话，细微的 200-300 字
- 每个部分后问用户是否看起来正确
- 覆盖：架构、组件、数据流、错误处理、测试
- 准备好回头澄清不理解的地方

### 在现有代码库中工作

- 提出变更前先探索当前结构。遵循现有模式
- 如果现有代码有问题影响当前工作（如文件过大、职责不清），将针对性改进作为设计的一部分
- 不要提出无关的重构。聚焦于服务当前目标的内容

## 设计之后

### 文档化

- 将验证过的设计（规格）写入 `docs/YYYY-MM-DD-<主题>-design.md`
- 提交设计文档到 git

### 规格自检

写完规格文档后，用新眼光检查：

1. **占位符扫描：** 有"TBD"、"TODO"、不完整部分、或模糊需求吗？修复它们。
2. **内部一致性：** 各部分之间有矛盾吗？架构与功能描述匹配吗？
3. **范围检查：** 聚焦程度够一个实施计划用吗，还是需要分解？
4. **歧义检查：** 有需求可以被两种方式解读吗？如果是，选一种并明确写出来。

发现问题就内联修复。不需要重新审阅——修复后继续。

### 用户审阅门控

规格审阅循环通过后，请用户审阅书面规格后再继续：

> "规格已写入并提交到 `<路径>`。请审阅，如有需要修改的地方请告诉我们，确认后再开始编写实施计划。"

等待用户回复。如果要求修改，修改后重新运行规格审阅循环。只有用户批准后才继续。

### 实施

- 调用 plan skill 创建详细实施计划
- **不要**调用任何其他技能。plan 是下一步。

## 关键原则

- **一次一个问题** — 不要用多个问题压倒用户
- **优先选择题** — 尽可能比开放性问题更容易回答
- **YAGNI 无情** — 从所有设计中移除不必要的功能
- **探索替代方案** — 在确定之前总是提出 2-3 种方案
- **增量验证** — 呈现设计，获得批准后再继续
- **灵活** — 不理解时回头澄清
