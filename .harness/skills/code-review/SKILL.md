# 代码审查 (Code Review)

> 来源：Superpowers (226k stars) — obra/superpowers
> 核心改编：适配团队协作流程，增加中文审查维度

## 核心原则

**早审查，勤审查。** 问题越晚发现，修复成本越高。

## 何时审查

**必须审查：**
- 完成一个独立任务后
- 实现重大功能后
- 合并到 main/master 前

**建议审查：**
- 卡住时（换个视角）
- 重构前（基线检查）
- 修复复杂 Bug 后

## 审查维度（按优先级）

### 1. 🔴 正确性（Critical）
- 逻辑错误、边界条件、空值处理
- 并发安全、竞态条件
- 错误处理是否完整

### 2. 🔴 安全性（Critical）
- 硬编码密钥、注入风险、权限漏洞
- 输入校验、输出转义
- 敏感数据泄露

### 3. 🟡 可维护性（Important）
- 命名是否清晰
- 职责是否单一
- 是否过度抽象或不够抽象
- 重复代码

### 4. 🟢 风格一致性（Minor）
- 是否遵循项目编码规范
- 注释是否充分
- 格式是否统一

## 审查流程

### 1. 获取变更范围

```bash
# 查看本次变更
git diff origin/main --stat
git diff origin/main
```

### 2. 逐维度审查

按上述 4 个维度依次检查，每个维度列出发现的问题。

### 3. 输出审查报告

```markdown
## 代码审查报告

### 变更概述
[一句话描述本次变更做了什么]

### 发现的问题

#### 🔴 Critical（必须修复）
- [文件:行号] [问题描述] [建议修复方式]

#### 🟡 Important（建议修复）
- [文件:行号] [问题描述] [建议修复方式]

#### 🟢 Minor（可选修复）
- [文件:行号] [问题描述]

### 优点
- [做得好的地方]

### 总结
[通过 / 需修改后通过 / 不通过]
```

### 4. 处理反馈

- **Critical** → 立即修复
- **Important** → 修复后再继续
- **Minor** → 记录，后续处理
- **审查者错了** → 用技术理由反驳，展示代码/测试证明

## 红旗

**永远不要：**
- 因为"很简单"就跳过审查
- 忽略 Critical 问题
- 带着 Important 问题继续推进
- 对有效的技术反馈争辩

## 与 verification.md 的关系

审查通过后，仍需运行 `.harness/verification.md` 的自验证流程。审查发现问题 ≠ 验证通过。
