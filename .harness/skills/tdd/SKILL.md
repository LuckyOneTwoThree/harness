# 测试驱动开发 (TDD)

> 来源：Superpowers (226k stars) — obra/superpowers
> 核心改编：保留 Iron Law，精简论证，适配多语言团队

## 铁律

```
没有失败测试，不准写生产代码
```

先写了代码再补测试？删掉，重来。没有例外：
- 不保留"参考"
- 不"适配"已有代码
- 删了就是删了

## 何时使用

**必须：** 新功能、Bug 修复、重构、行为变更
**可跳过（需人类确认）：** 一次性原型、生成代码、配置文件

想"这次就跳过 TDD"？停。那是自我合理化。

## 红-绿-重构

### 🔴 RED — 写失败测试

写一个最小测试，展示期望行为：

```python
# ✅ 好：清晰命名，测试真实行为，一个断言
def test_rejects_empty_email():
    result = submit_form(email="")
    assert result.error == "Email required"

# ❌ 坏：模糊命名，测试 mock 而非代码
def test_form_works():
    mock = Mock()
    submit_form(mock)
    mock.assert_called_once()
```

**要求：** 一个行为、清晰命名、尽量用真实代码（非 mock）

### 验证 RED — 必须看它失败

```bash
pytest tests/test_form.py::test_rejects_empty_email -v
```

确认：
- 测试 **失败**（不是报错）
- 失败原因是功能缺失（不是拼写错误）
- 测试通过了？说明你在测试已有行为，改测试

### 🟢 GREEN — 最小代码

写最简单的代码让测试通过：

```python
def submit_form(email: str) -> Result:
    if not email.strip():
        return Result(error="Email required")
    # ...
```

不要加功能、不要重构其他代码、不要"改进"超出测试范围。

### 验证 GREEN — 必须看它通过

```bash
pytest tests/test_form.py -v
```

确认：测试通过 + 其他测试未被破坏 + 输出干净（无警告）

### 🔵 REFACTOR — 清理

只在绿灯后：
- 消除重复
- 改善命名
- 提取辅助函数

保持测试绿灯，不加新行为。

### 重复

下一个失败测试 → 下一个功能。

## 常见借口 vs 现实

| 借口 | 现实 |
|------|------|
| "太简单不用测" | 简单代码也会坏，测试只要 30 秒 |
| "先写代码再补测试" | 后补的测试立刻通过，证明不了任何事 |
| "删掉 X 小时的代码太浪费" | 沉没成本谬误。保留不可信的代码才是技术债 |
| "TDD 太教条了" | TDD 就是务实：比事后 debug 更快 |
| "手动测过了" | 手动 ≠ 系统化。无法复现，无法回归 |

## 红旗 — 停下来重来

- 先写了代码再写测试
- 测试立刻通过
- 无法解释测试为什么失败
- 说"这次不一样"
- 说"保留作参考"

**所有这些 = 删掉代码，从 TDD 重新开始。**

## Bug 修复流程

发现 Bug → 写失败测试复现它 → 遵循 TDD 循环 → 测试证明修复且防止回归。

**永远不要在没有测试的情况下修 Bug。**

## 完成检查清单

- [ ] 每个新函数/方法都有测试
- [ ] 每个测试都看过它失败
- [ ] 每个测试都因预期原因失败（功能缺失，非拼写错误）
- [ ] 写了最小代码让测试通过
- [ ] 所有测试通过
- [ ] 输出干净（无错误、无警告）
- [ ] 测试用真实代码（mock 仅在不可避免时）
- [ ] 边界条件和错误场景已覆盖

全部打勾才能标记完成。漏了任何一项 = 跳过了 TDD，重来。
