# 编写实施计划 (Writing Plans)

> 来源：Superpowers (226k stars) — obra/superpowers + Harper Reed 工作流
> 核心改编：适配我们的 features.json + progress.md 体系

## 核心原则

假设执行者对代码库零上下文。文档化他们需要知道的一切：改哪些文件、怎么测试、怎么验证。

## 何时使用

- 收到复杂任务（3+ 步骤）
- 需求涉及多个模块
- 不确定最佳实现路径
- 需要人类确认方案后再动手

## 计划结构

### 头部

```markdown
# [功能名称] 实施计划

**目标:** [一句话描述做什么]
**架构:** [2-3 句话描述方案]
**技术栈:** [关键技术/库]
**预估时间:** [可选]
```

### 文件映射

在定义任务之前，先映射涉及的文件：

```markdown
## 涉及文件

| 操作 | 文件 | 职责 |
|------|------|------|
| 新建 | `src/services/auth.py` | 认证逻辑 |
| 修改 | `src/api/routes.py:45-67` | 添加登录端点 |
| 测试 | `tests/test_auth.py` | 认证测试 |
```

### 任务分解

每个任务是一个可独立理解和验证的单元：

```markdown
## Task 1: [组件名称]

**涉及文件:**
- 新建: `exact/path/to/file.py`
- 修改: `exact/path/to/existing.py:123-145`
- 测试: `tests/exact/path/to/test.py`

- [ ] **Step 1: 写失败测试**
  [具体测试代码]

- [ ] **Step 2: 运行测试确认失败**
  命令: `pytest tests/path/test.py::test_name -v`
  期望: FAIL with "xxx not defined"

- [ ] **Step 3: 写最小实现**
  [具体实现代码]

- [ ] **Step 4: 运行测试确认通过**
  命令: `pytest tests/path/test.py::test_name -v`
  期望: PASS

- [ ] **Step 5: 提交**
  ```bash
  git commit -m "feat(scope): description"
  ```
```

## 任务粒度

**每步是一个动作（2-5 分钟）：**
- "写失败测试" — 一步
- "运行确认失败" — 一步
- "写最小实现" — 一步
- "运行确认通过" — 一步
- "提交" — 一步

## 禁止事项

这些是**计划失败**，永远不要写：
- "TBD"、"TODO"、"待实现"
- "添加适当的错误处理"（没有具体代码）
- "为上面的代码写测试"（没有实际测试代码）
- "类似 Task N"（重复代码——执行者可能乱序阅读）
- 只描述做什么但不展示怎么做（代码步骤必须有代码块）

## 自检清单

写完计划后，用新眼光检查：

1. **需求覆盖** — 每个需求都能指向一个任务？列出遗漏。
2. **占位符扫描** — 搜索"待定"、"TODO"等红旗，修复。
3. **类型一致性** — 后续任务中的类型、方法名、属性名与前面定义的一致？

## 执行方式

计划完成后，与用户确认执行方式：

1. **逐步执行** — 每个任务完成后暂停，等用户确认再继续
2. **连续执行** — 按计划逐个完成，全部做完后汇报

推荐逐步执行，特别是复杂功能。

## 与 features.json 的关系

将计划中的任务同步到 `.harness/features.json`：
- 每个 Task 对应一个 feature 条目
- 完成后更新状态为 `done`
- 阻塞时更新为 `blocked` 并记录原因
