# 完成开发分支

> 来源：Superpowers (228k stars) — obra/superpowers/skills/finishing-a-development-branch
> 原文链接：https://github.com/obra/superpowers/blob/main/skills/finishing-a-development-branch/SKILL.md
> 改编：简化 worktree 相关逻辑（团队暂不使用 worktree）

通过呈现清晰的选项并处理选定的工作流，引导开发工作的完成。

**核心原则：** 验证测试 → 检测环境 → 呈现选项 → 执行选择 → 清理。

## 流程

### 步骤 1：验证测试

**在呈现选项之前，验证测试通过：**

```bash
# 运行项目测试套件
npm test / pytest / go test ./...
```

**测试失败：**
```
测试失败（N 个失败）。必须先修复才能继续：
[展示失败]
在测试通过之前无法进行合并/PR。
```
停止。不要进入步骤 2。

**测试通过：** 继续步骤 2。

### 步骤 2：确定基础分支

```bash
# 尝试常见基础分支
git merge-base HEAD main 2>/dev/null || git merge-base HEAD master 2>/dev/null
```

或询问："这个分支从 main 拉出的——对吗？"

### 步骤 3：呈现选项

**精确呈现以下 4 个选项：**

```
实现完成。你想怎么做？

1. 合并回 <基础分支>（本地）
2. 推送并创建 Pull Request
3. 保持分支现状（我稍后处理）
4. 丢弃这个工作

选哪个？
```

不要添加解释——保持选项简洁。

### 步骤 4：执行选择

#### 选项 1：本地合并

```bash
git checkout <基础分支>
git pull
git merge <功能分支>

# 在合并结果上验证测试
<测试命令>

# 合并成功后：删除分支
git branch -d <功能分支>
```

**必须在合并结果上验证测试后才能删除分支。**

#### 选项 2：推送并创建 PR

```bash
git push -u origin <功能分支>

gh pr create --title "<标题>" --body "$(cat <<'EOF'
## 概要
<2-3 个要点说明改了什么>

## 测试计划
- [ ] <验证步骤>
EOF
)"
```

**不要清理分支** — 用户需要它来处理 PR 反馈。

#### 选项 3：保持现状

报告："保持分支 <名称>。"

#### 选项 4：丢弃

**先确认：**
```
这将永久删除：
- 分支 <名称>
- 所有提交：<提交列表>

输入 'discard' 确认。
```

等待精确确认。确认后：
```bash
git branch -D <功能分支>
```

## 快速参考

| 选项 | 合并 | 推送 | 清理分支 |
|------|------|------|----------|
| 1. 本地合并 | ✅ | — | ✅ |
| 2. 创建 PR | — | ✅ | — |
| 3. 保持现状 | — | — | — |
| 4. 丢弃 | — | — | ✅ (force) |

## 常见错误

| 错误 | 问题 | 修复 |
|------|------|------|
| 跳过测试验证 | 合入坏代码 | 选项前必须验证测试 |
| 开放式问题 | "接下来做什么？"太模糊 | 精确呈现 4 个选项 |
| 选项 2 清理分支 | 删除了用户处理 PR 反馈需要的分支 | 只有选项 1 和 4 清理分支 |
| 丢弃不确认 | 误删工作 | 要求输入 'discard' 确认 |

## 红旗

**永远不要：**
- 测试失败时继续
- 未在合并结果上验证测试就合并
- 不确认就删除工作
- 未明确请求就 force push

**必须：**
- 选项前验证测试
- 呈现精确 4 个选项
- 选项 4 要求输入确认
- 只有选项 1 和 4 清理分支
