# 交付前验证 (Verification Before Completion)

> 来源：Superpowers (226k stars) — obra/superpowers
> 核心改编：与我们的 verification.md 互补，聚焦"证据优先"

## 铁律

```
没有新鲜验证证据，不准声称完成
```

## 验证门控

```
在声称任何状态或表达满意之前：

1. 识别：什么命令能证明这个声明？
2. 运行：执行完整命令（全新、完整）
3. 读取：完整输出，检查退出码，计数失败
4. 确认：输出是否证实了声明？
   - 否 → 用证据说明实际状态
   - 是 → 带着证据做声明
5. 只有这时：做声明

跳过任何一步 = 说谎，不是验证
```

## 常见失败对照

| 声明 | 需要 | 不够 |
|------|------|------|
| 测试通过 | 测试命令输出：0 失败 | 上次跑过、"应该通过" |
| Linter 干净 | Linter 输出：0 错误 | 部分检查、推断 |
| 构建成功 | 构建命令：exit 0 | Linter 通过、日志看起来正常 |
| Bug 已修复 | 测试原始症状：通过 | 改了代码、假设修好了 |
| 需求已满足 | 逐条检查清单 | 测试通过 |

## 红旗 — 停下来

- 使用"应该"、"可能"、"看起来"
- 验证前表达满意（"搞定了！"、"完美！"）
- 准备 commit/push/PR 但没跑验证
- 信任 Agent 的成功报告
- 依赖部分验证
- **任何暗示成功但没跑验证的措辞**

## 常见借口

| 借口 | 现实 |
|------|------|
| "应该没问题了" | 跑验证 |
| "我有信心" | 信心 ≠ 证据 |
| "这次就免了" | 没有例外 |
| "Linter 过了" | Linter ≠ 编译器 |
| "Agent 说成功了" | 独立验证 |
| "累了想收工" | 疲惫 ≠ 借口 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到: 34/34 通过] "所有测试通过"
❌ "应该没问题了" / "看起来对"
```

**构建：**
```
✅ [运行构建] [看到: exit 0] "构建通过"
❌ "Linter 过了"（Linter 不检查编译）
```

**需求：**
```
✅ 重读计划 → 创建检查清单 → 逐条验证 → 报告差距或完成
❌ "测试过了，阶段完成"
```

## 与 verification.md 的关系

本 skill 是 `.harness/verification.md` 的补充：
- **verification.md** 定义了完整的自验证协议和失败处理流程
- **本 skill** 聚焦于"证据优先"的思维模式——在做任何声明之前必须有验证输出

两者配合使用：本 skill 确保你不会虚假声明完成，verification.md 确保失败时有正确的处理流程。

## PRD 验收标准映射

如果当前任务关联了 PRD（features.json 中有 `context_ref`），verify 必须额外执行：

### 1. 提取 PRD 验收标准

从 PRD 的 `## 7. 验收标准` 中提取所有 AC-xxx 编号和对应描述。

### 2. 检查测试覆盖

在测试代码中搜索 AC-xxx 标记：
```bash
# 搜索测试文件中的 AC 编号引用
grep -rn "AC-[0-9]*" tests/ --include="*.py" --include="*.ts" --include="*.go"
```

### 3. 生成验收报告

将测试结果映射到 PRD 验收标准，生成验收报告：

```markdown
## 验收报告

PRD: [PRD 文件路径]
PRD 版本: [版本号]
验收日期: [日期]

| 编号 | 验收标准 | 测试方式 | 结果 | 证据 |
|------|----------|----------|------|------|
| AC-001 | [标准描述] | 自动 | ✅ 通过 | [测试文件:用例名] |
| AC-002 | [标准描述] | 自动 | ✅ 通过 | [测试文件:用例名] |
| AC-003 | [标准描述] | — | ❌ 无对应测试 | 需要补充 |

总结: X/Y 通过，Z 条无对应测试
```

### 4. 测试命名约定

为确保验收标准可追溯，测试用例命名必须包含 AC 编号：
```python
def test_AC001_phone_login():
    """AC-001: 支持手机号+验证码登录"""
    ...
```

### 5. 保存验收报告

报告保存到 `.harness/context/acceptance/AR-{PRD编号}-{日期}.md`
