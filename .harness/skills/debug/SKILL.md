# 系统化调试 (Systematic Debugging)

> 来源：Superpowers (226k stars) — obra/superpowers
> 核心改编：保留四阶段流程，精简论证

## 铁律

```
没有根因分析，不准提修复方案
```

随意修复浪费时间，还会引入新 Bug。快速补丁掩盖根本问题。

## 四个阶段

必须按顺序完成每一阶段，才能进入下一阶段。

### 阶段 1：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误和警告
   - 错误信息通常包含解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 能可靠触发吗？步骤是什么？
   - 每次都发生吗？
   - 无法复现 → 收集更多数据，不要猜

3. **检查最近变更**
   - 什么改动可能导致这个问题？
   - `git diff`、最近提交
   - 新依赖、配置变更、环境差异

4. **多组件系统：收集证据**
   ```
   对每个组件边界：
   - 记录进入组件的数据
   - 记录离开组件的数据
   - 验证环境/配置传递
   
   运行一次收集证据 → 分析哪个组件出问题 → 针对性调查
   ```

5. **追踪数据流**
   - 错误值从哪来？
   - 谁用错误值调用了这里？
   - 向上追踪到源头
   - 在源头修复，不在症状处修复

### 阶段 2：模式分析

1. **找同类工作的代码** — 代码库里有什么类似的正常工作的？
2. **对比差异** — 工作的和坏的有什么不同？列出每个差异
3. **理解依赖** — 需要什么组件、配置、环境？做了什么假设？

### 阶段 3：假设与验证

1. **形成单一假设** — "我认为 X 是根因，因为 Y"
2. **最小变更验证** — 只改一个变量
3. **验证结果** — 有效 → 阶段 4；无效 → 新假设（不要叠加修复）
4. **不知道就说不知道** — 不要假装知道，寻求帮助

### 阶段 4：实施修复

1. **写失败测试** — 最小复现，自动化测试优先
2. **实施单一修复** — 只修根因，不做"顺手改进"
3. **验证修复** — 测试通过？其他测试未破坏？
4. **修复失败？**
   - 尝试 < 3 次 → 回到阶段 1，用新信息重新分析
   - **尝试 ≥ 3 次 → 停止，质疑架构**（见下文）

### 3 次修复失败 = 架构问题

**模式：** 每次修复揭示新的耦合/状态问题、修复在别处产生新症状。

**停下来问：**
- 这个模式从根本上合理吗？
- 是不是靠惯性在坚持？
- 应该重构架构还是继续修症状？

**与人类讨论后再继续。** 这不是假设失败——是架构错误。

## 红旗 — 停下来回到阶段 1

如果你在想：
- "先随便改改看"
- "加多个改动一起跑测试"
- "跳过测试，手动验证"
- "大概率是 X，先修了再说"
- "我不完全理解但可能有效"
- **"再试一次"（已经试了 2+ 次）**

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| 1. 根因 | 读错误、复现、检查变更、收集证据 | 理解 WHAT 和 WHY |
| 2. 模式 | 找工作示例、对比差异 | 识别不同点 |
| 3. 假设 | 形成理论、最小验证 | 确认或新假设 |
| 4. 实施 | 写测试、修复、验证 | Bug 解决、测试通过 |

## 实际效果

- 系统化调试：15-30 分钟修复
- 随意修复：2-3 小时折腾
- 首次修复成功率：95% vs 40%
- 引入新 Bug：接近零 vs 常见
