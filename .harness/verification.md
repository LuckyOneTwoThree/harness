# AI 编程助手自我验证协议 (Self-Verification Protocol)

作为 AI Agent，你的终极目标是**交付经过验证的工作，而非仅仅是生成代码**。在修改完代码后，你必须执行以下验证闭环。

## 🟢 第一阶段：交付前必做 (Pre-Delivery)

在你决定更新 `features.json` 中的状态为 `done` 或通知人类工作完成之前，**必须**依次完成：

1. **Lint 检查** — 运行项目配置的代码质量工具，确保零错误。
2. **测试通过** — 运行相关单元测试 / 集成测试，确保全部通过。
3. **安全扫描** — 运行 `.harness/scripts/security-check.sh`，确保零错误。
4. **记录结果** — 将以上验证结果（通过/失败 + 输出摘要）记录到 `.harness/progress.md`。

> 以上检查由 CI 门控自动执行。即使你跳过本地验证，CI 也会拦截。
> features.json 中标记为 "done" 的功能必须有对应测试，CI 会检查。

## 🔴 第二阶段：失败降级与防死锁机制 (The "Doom Loop" Breaker)

如果脚本报错，或者你发现代码运行不如预期，请**严格遵守**以下重试与求助法则：

1. **第一次失败 (Attempt 1)**：自行阅读错误日志，分析原因，实施**第一种**修复策略，并记录在 `progress.md` 中。
2. **第二次失败 (Attempt 2)**：如果你采取了策略 A 但依旧报类似错误，**禁止在策略 A 上继续微调**！你必须尝试一条**完全不同**的技术路径（策略 B）。
3. **第三次失败 (Attempt 3 = STOP)**：如果你尝试了两种不同的路径均告失败，说明你缺少某些隐藏上下文或系统限制超出了你的认知。
   👉 **此时你必须立即停止，并将控制权交还给人类。**

### 向人类报告的格式：

当你触发 Attempt 3 停止规则时，必须输出如下格式报告给人类：

🚨 **自我验证失败，请求人类介入**
- **目标**: [你试图完成的任务]
- **已尝试策略 A**: [简述，以及为什么失败]
- **已尝试策略 B**: [简述完全不同的策略 B，以及为什么失败]
- **报错摘要**: [最近一次的 Error Log 摘要]
- **可能缺失的上下文**: [你猜测人类需要提供什么信息，比如：是否缺少某个依赖库？是否需要开通网络权限？]

## CI 门控集成

CI 会自动执行验证检查：
1. Lint 检查 → 不通过则阻断 PR
2. 测试套件 → 不通过则阻断 PR
3. 安全扫描 → 有错误则阻断 PR
4. features.json 检查 → 标记 "done" 但无对应测试的条目会触发警告

**Agent 无法绕过这些门控。即使 Agent 在 Prompt 层面被说服跳过验证，CI 层也会拦截。**

## 验证示例

以下是一个完整的验证流程示例，展示 Agent 说"做完了"时应该怎么做：

```
Agent: "登录功能已完成。"

验证门控触发:
1. 识别：什么命令能证明登录功能正常？
   → pytest tests/test_auth.py -v

2. 运行：
   $ pytest tests/test_auth.py -v
   === 12 passed, 0 failed ===

3. 读取：
   - 退出码: 0
   - 通过: 12
   - 失败: 0
   - 警告: 0

4. 确认：输出证实了"登录功能已完成"的声明
   → ✅ 验证通过

5. 声明：
   "登录功能已完成。证据：pytest 12/12 通过，无警告。"
```

**反面示例（不允许）：**
```
Agent: "登录功能应该没问题了。" ← 没有运行验证
Agent: "代码看起来是对的。" ← 没有证据
Agent: "我检查过了，都通过了。" ← 没有粘贴输出
```
