# 通用编码规范 (General Coding Standards)

> 本文件为所有项目的基线规范。语言级规范见 `languages/` 目录，按项目主语言加载。
> 当本规范与语言规范冲突时，**语言规范优先**（语言规范更具体）。
> 当两条规则矛盾时（如性能 vs 可读性），优先级：安全 > 正确性 > 可读性 > 性能。
>
> **兜底规则：** 如果项目使用的语言在 `languages/` 中没有专项规范，Agent 应：
> 1. 遵循该语言的官方 Style Guide（如 PEP 8、Effective Go、Google Style Guide）
> 2. 工具链选择社区最主流的方案
> 3. 将选择记录到项目级 AGENTS.md 中，以便后续统一

## 命名
- 变量和函数名称必须具有自解释性，避免 `temp`, `data`, `res` 等模糊命名。
- 常量用 UPPER_SNAKE_CASE，类用 PascalCase，其余跟随语言惯例。
- 布尔变量用 `is_` / `has_` / `can_` 前缀（如 `is_active`、`has_permission`）。

## 注释
- 公共函数、类必须写 Docblock/Docstring。
- 解释 **为什么 (Why)**，不解释 **是什么 (What)**。
- TODO 格式：`// TODO(username): 具体描述`
- 复杂算法必须有注释，说明思路和时间/空间复杂度。

## 错误处理
- 严禁静默吞没异常（空 `catch` 块）。
- 必须捕获具体异常类型，避免裸 `Exception`。
- 错误信息必须包含上下文参数（操作 + 原因 + 建议）。
- 区分业务异常（可恢复，返回错误码）和系统异常（需告警，返回 500）。

## 日志
- `DEBUG` — 调试信息，生产环境关闭
- `INFO` — 业务关键状态变更（用户登录、订单创建）
- `WARN` — 不影响主流程的异常（重试、降级）
- `ERROR` — 功能失败，附带 Stack Trace
- 日志中禁止输出完整密钥、密码、Token（最多前 4 位 + `****`）

## 代码质量
- 函数不超过 50 行，超过则拆分。
- 文件不超过 300 行，超过则模块化。
- 嵌套不超过 3 层，用 early return 降级。
- 圈复杂度不超过 10。

## 精准改动（Karpathy 第三原则）
- **只改该改的**。每一行改动都能直接追溯到用户请求。
- 不要"顺手改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。匹配现有风格，即使你会写得不同。
- 发现无关死代码：提一下，不要删。
- 你的改动产生的孤儿代码（未使用的 import/变量/函数）：清理。
- 已有的死代码：不清理，除非被要求。

## 测试
- 新功能必须有测试，Bug 修复必须有回归测试。
- 覆盖率不低于 70%（可按项目调整）。
- 测试命名：`test_<功能>_<场景>_<期望>`
- **AC 编号约定：** 如果任务关联了 PRD，测试用例命名必须包含 AC 编号：`test_AC001_<描述>()`。这确保验收标准可追溯，verify skill 能自动映射测试结果到 PRD 验收标准。
- 测试必须隔离：不依赖外部服务、不依赖执行顺序、不共享可变状态。
- Mock 外部依赖（数据库、第三方 API、文件系统），不 Mock 被测对象本身。

## 依赖
- 不引入非必要依赖。引入前评估：star 数、维护频率、安全漏洞、License 兼容性。
- 锁定版本号，禁止 `*` 或 `latest`。
- 使用 lockfile（`package-lock.json` / `poetry.lock` / `go.sum`）并纳入版本控制。

## API 设计
- 统一响应格式：`{ code, message, data }` 或项目约定的格式。
- 错误码用数字或枚举，不要用魔法字符串。
- API 版本化：URL 路径（`/v1/`）或 Header，按项目约定。
- 分页必须有 `page` / `size` 或 `cursor` 参数，禁止返回无限列表。
- 敏感接口必须有速率限制（rate limiting）。

## 文档
- 每个项目必须有 README.md：项目简介、快速开始、环境要求、部署方式。
- API 变更必须同步更新 API 文档（Swagger/OpenAPI 或项目约定格式）。
- 重大变更记录到 CHANGELOG.md。

## 代码审查关注点（Agent 生成的代码也需要 Review）
1. **安全性**：有无硬编码密钥、注入风险、权限漏洞
2. **正确性**：边界条件、空值处理、并发安全
3. **可维护性**：命名清晰、职责单一、不过度抽象
4. **测试覆盖**：关键路径有测试、边界有测试
