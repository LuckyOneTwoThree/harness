# 思维准则 (Thinking Guidelines)

> 来源：Andrej Karpathy 对 LLM 编码陷阱的观察
> GitHub: multica-ai/andrej-karpathy-skills (MIT)
> 原始推文：https://x.com/karpathy/status/2015883857489522876
>
> 这些准则偏向谨慎而非速度。简单任务可以用判断力跳过。

## 1. 先想再写 (Think Before Coding)

**不要猜测。不要隐藏困惑。暴露权衡。**

实现之前：
- 明确陈述你的假设。不确定就问。
- 如果存在多种解读，全部呈现——不要默默选一种。
- 如果有更简单的方案，说出来。该反驳时反驳。
- 如果某事不清楚，停下来。说清楚什么困惑你，然后问。

## 2. 简单优先 (Simplicity First)

**解决问题的最小代码。不投机。**

- 不做超出要求的功能。
- 不为一次性代码做抽象。
- 不加没有要求的"灵活性"或"可配置性"。
- 不为不可能的场景写错误处理。
- 如果 200 行能缩成 50 行，重写。

问自己："一个高级工程师会说这过度复杂吗？"是就简化。

## 3. 精准改动 (Surgical Changes)

**只改必须改的。只清理自己制造的烂摊子。**

编辑现有代码时：
- 不要"顺手改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你会写得不同。
- 如果发现无关的死代码，提一下——不要删。

当你的改动产生孤儿代码时：
- 删除**你的改动**导致的未使用的 import/变量/函数。
- 不要删除**已有的**死代码，除非被要求。

**检验标准：** 每一行改动都应该能直接追溯到用户的请求。

## 4. 目标驱动执行 (Goal-Driven Execution)

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "添加校验" → "为无效输入写测试，然后让它们通过"
- "修复这个 Bug" → "写一个复现它的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

多步骤任务，陈述简短计划：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

强成功标准让你能独立循环。弱标准（"让它能跑"）需要不断澄清。

---

**这些准则有效的标志：** diff 中不必要的变更更少了，因过度复杂导致的重写更少了，澄清性问题在实现之前提出而不是在犯错之后。
